## Important caveat: CV hourly cap not exercised

Going in, the goal was to cross CV's 200/hr cap and exercise simyan's
`pyrate_limiter` hourly-stall path (a different code path from mokkari's
raise-and-retry that we hadn't validated). The run **did not** reach this state,
for two compounding reasons:

1. **Metron is the binding constraint.** Metron's 20/min cap is much tighter
   than CV's 60/min. Wall time is paced by Metron, so CV gets a long observation
   window for the same number of requests. Over the 59-min run, even 200+ CV
   requests would have averaged well under 200/hr.

2. **Most fixtures have stored CV IDs** (already tagged in prior runs). Those
   take the cheap "refresh via stored id" code path — a single CV request per
   fixture. The 100 fixtures yielded only 128 CV requests (1.28 req/fixture
   average), nowhere near the ~6 req/fixture cost of a full search.

Empirical scaling: doubling fixtures to ~200 doubles Metron cost but also
doubles the time window, so CV's hourly rate stays roughly constant. Under the
**natural unattended workflow**, CV's hourly cap is unreachable — Metron paces
the run too slowly.

**To actually exercise CV's hourly cap, a focused follow-up run is needed:**

- `--sources comicvine --force-search --limit 100 --jobs 8` — single- source
  removes Metron pacing; `--force-search` bypasses the refresh-by-id shortcut.
  ~6 CV req/fixture × 100 fixtures = 600 requests, comfortably above the 200/hr
  cap.
- Estimated wall time: ~2-3 hours once `pyrate_limiter` starts stalling.

I'm adding this as a separate TODO item below rather than launching it inline;
the user may want to schedule it overnight or skip it based on cost/value
judgment.

## Retry-exhaustion cascades at scale

The 16 WARNINGs in the 20-fixture run scaled to **551 WARNINGs** in the
100-fixture run — non-linear because retry exhaustion cascades on high-fan-out
fixtures. Breakdown:

- **86 × `metron: series search failed`** — the un-retried `Metron.series_list`
  call at `metron.py:254`. Known issue from the prior run. Counts roughly
  linearly (100 vs 20 fixtures).
- **~465 × `metron: issue-list for series N failed`** — per-candidate-series
  retry exhaustion (`_MAX_RATE_LIMIT_RETRIES = 5` too small under -j 8).
  Distribution is HEAVILY concentrated on high-fan-out fixtures — 32 distinct
  series accounted for these, with a few series fielding 30+ failures (Conan
  series in particular, where a single fixture fans out to 20+ candidate series
  in the Metron database).

The cascade pattern: a Conan-titled fixture's series search returns 20+ "Conan
..." series. Each candidate spawns an `_issues_list_with_retry` call. Under -j 8
contention they all hit the 20/min Metron cap simultaneously, exhaust their 5
rate-limit retries (each), and log WARNINGs. Net effect for that one fixture:
maybe 5 of 20 candidate series actually return their issues; the rest are
silently dropped. The matcher proceeds with a partial candidate set.

This is the same issue surfaced in the 20-fixture run but quantified at scale.
The follow-up to bump `_MAX_RATE_LIMIT_RETRIES` or make it `-j`-aware is more
urgent than I'd weighted it before — high-fan-out fixtures lose meaningful
candidate coverage at -j 8.

## What this run does prove

1. **At-scale rate-limiter compliance.** 995 Metron + 128 CV requests over 59
   min, both within caps, no rate-limiter violations.
2. **No deadlocks at scale.** 100 fixtures across 8 workers completed cleanly.
   Process-wide buckets serialise correctly.
3. **No exceptions, no crashes.** Exit 0, zero tracebacks.
4. **Retry-exhaustion behaviour is graceful**: WARNINGs, partial candidate sets,
   `continue` past failed series. No corrupted writes, no aborted runs.

## What it doesn't prove

- **CV pyrate_limiter hourly-stall behaviour.** Unreached this run. See
  follow-up below.
- **Tagging quality under degradation.** With 32 series suffering partial
  candidate-set drops, some matches almost certainly went to the wrong candidate
  or got skipped. The matcher's correctness numbers under degraded conditions
  need separate measurement — arguably the most user-visible consequence of -j 8
  under cold- cache contention.

## Follow-ups added by this run

1. **CV-only force-search stress run.** As described above. The only way to
   actually cross the 200/hr cap and validate simyan's blocking behaviour.
2. **`_MAX_RATE_LIMIT_RETRIES` bump.** Promoted from "worth measuring" to
   "needed before -j 8 becomes a reasonable default." 5 is too small for
   high-fan-out series in concurrent runs.
3. **Tagging-quality measurement under -j 8 cold cache.** New work: re-run a
   labeled calibration set at -j 8 vs -j 1, diff the outcomes, quantify how much
   accuracy the parallel path costs us. This is the test that matters most for
   user-visible behaviour.
