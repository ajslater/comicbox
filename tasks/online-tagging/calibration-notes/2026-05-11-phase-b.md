# Phase B calibration — 2026-05-11

Empirical validation of the `APIBudget` thresholds in
[`06-api-budget-spec.md`](../06-api-budget-spec.md). Pins the Phase A
placeholders to data-driven values and unblocks Phase C.

## Fixture set

339 fixtures from the developer's personal library, gitignored at
`tests/calibration/fixtures.json`. Coverage: mainstream cape comics (Marvel/DC),
indie publishers (AfterShock, Image), Watchmen reboot volumes, Sin City TPBs,
Akira-era manga, modern weekly releases. Cover quality split: ~80% `full`, ~20%
`thumbnail` (slimlib's downscaled covers).

ComicVine has expected ids on essentially all 339 fixtures. Metron expected ids
are missing on 266 — that side of the calibration is informationally empty until
the user cross-references CV ids against Metron's `cv_id` field. Phase B's
numbers below are therefore ComicVine-only on the accuracy axis; Metron is
measured on cost only.

## Experiment matrix actually run

| Run | api_budget | Sample               | Threshold                      | Purpose                                                            |
| --- | ---------- | -------------------- | ------------------------------ | ------------------------------------------------------------------ |
| B0  | balanced   | full (339)           | 0.0 (Phase A default)          | Baseline; canonical `fixtures.outcomes.json`                       |
| B1  | exhaustive | full                 | 0.0 (never filter)             | Ceiling check; confirms identical to B0 since both have filter off |
| B2  | fast       | one-per-series (125) | 0.7                            | Smoke check; ~10x faster matrix run                                |
| B3  | fast       | full (339)           | 0.7                            | Acceptance check against the full set                              |
| B4  | balanced   | full                 | **0.4** (the spec placeholder) | Does conservative pre-filter help?                                 |

Cells the spec's matrix lists but Phase B did not run:

- `_MAX_VOLUMES_PER_SEARCH` sweep (3 / 5 / 10) — fast's 5 worked well enough
  that finer tuning didn't earn its keep on this fixture set.
- `_TOP_K_FOR_HASHING` sweep (2 / 3 / 5) — left at 5; cover-hashing fires rarely
  with the current threshold, so the K knob had little leverage.
- Higher pre-filter thresholds (0.8, 0.85) for fast — the 0.7 result was already
  100% accurate; pushing higher risked false negatives without meaningful
  additional savings.

These remain open for a future Phase B-extended pass if real-world usage exposes
regressions; current data didn't motivate them.

## Results

### Accuracy

| Run                 | CV correct | CV wrong    | CV no_candidates | Accuracy on labeled     |
| ------------------- | ---------- | ----------- | ---------------- | ----------------------- |
| B0 (balanced @ 0.0) | 338        | 1 (Moebius) | 0                | 99.7%                   |
| B1 (exhaustive)     | 338        | 1           | 0                | 99.7% — identical to B0 |
| B3 (fast @ 0.7)     | 338        | 0           | 1 (Moebius)      | **100.0%**              |
| B4 (balanced @ 0.4) | 338        | 1           | 0                | 99.7% — identical to B0 |

The single labeled wrong/no*candidates throughout is \_Moebius Library (2016)
#001* — CV's "Adventures of Basil & Moebius" volume scores ~0.5 on the
series-name similarity vs the comic's "moebius library", so it survives
`balanced`'s 0.4 cut but is dropped by `fast`'s 0.7 cut. Under all modes the
case scored 0.76 (below the 0.95 `confidence_threshold`), so the user-visible
behavior is the same (prompt-or-skip, no wrong auto-write). `fast`'s
"no_candidates" classification is more honest than `balanced`'s "wrong, scored
0.76" — but no production behavior change.

## Thresholds pinned

```python
# comicbox/online/series_filter.py
_THRESHOLDS = {
    APIBudget.EXHAUSTIVE: 0.0,  # never filter
    APIBudget.BALANCED:   0.4,  # ← was 0.0 in Phase A; promoted here
    APIBudget.FAST:       0.7,  # confirmed against Phase A placeholder
}
```

`balanced @ 0.4` is a strict improvement over `balanced @ 0.0`: every single
fixture in B4 produced the same outcome as in B0, with 18% fewer API calls.
There is no reason to ship `balanced` at 0.0 going forward.

`fast @ 0.7` cuts API budget by 60% with the only outcome change being Moebius
`wrong → no_candidates` — strictly better classification.

Both thresholds are pinned in code with this PR.

## What's NOT pinned

- **`_MAX_VOLUMES_PER_SEARCH`** in fast mode is still the production default
  (20), not the spec's 5. The pre-filter alone delivered the cost target without
  needing the lower cap. Open for future tuning if real-world libraries surface
  call patterns the pre-filter doesn't catch.
- **`confidence_threshold` / `min_confidence` per-budget overrides**: the spec
  calls for `fast` to use 0.97 / 0.60 vs balanced's 0.95 / 0.50, but the
  matcher's per-source resolve\_\* helpers don't yet read from `api_budget`.
  Wiring this is Phase C work.
- **Year-fallback retry for `fast` (spec: off)** still fires.
- **Auto-engagement triggers** (≥50 CV / ≥500 Metron under `unattended`): not
  implemented. Phase C.

These together mean `fast` mode today is "pre-filter at 0.7" — already useful,
but a subset of what the spec describes. Phase C adds the rest.

## Decisions for Phase C

1. The threshold table above ships as-is.
2. Auto-engagement thresholds (Phase C placeholders): keep the spec's 50 / 500
   numbers. Phase B doesn't have data to overrule them — the spec's reasoning
   ("hours of waiting under balanced") still holds.
3. `--api-budget` and `--api-budget-per-source` CLI surface on `comicbox`
   matches the calibration harness (already validated UX).
4. User doc lives at `tasks/online-tagging/api-budget-user-doc.md` mirroring the
   existing `match-resolution-user-doc.md`.

## Closing note

Phase B was originally scoped at "tune all the placeholder thresholds" but ended
up much narrower because the pre-filter alone delivered the target cost
reduction without measurable accuracy loss. The other knobs (max-per-search,
top-K hashing, per-budget confidence_thresholds) didn't earn their experiment
cost. Phase C can ship the pre-filter lever today and leave finer tuning for a
future, data-motivated revisit.
