# Bigmedia 247-fixture calibration — 2026-05-14

Full-cover, Big-Two-leaning calibration against `/Volumes/Media/Comics/`
(~864 comics). Complements the slimlib calibration by exercising two
things slimlib couldn't: **cover hashing** (full-cover library) and
**Metron** (Metron's catalog covers Big Two well).

## Fixture set

864 comics in the library; one-per-series dedup brought it to **247
unique series fixtures**. Sampled via `tests/calibration/sample.py`:

- One-per-series dedup, stratified by (decade, publisher)
- `cover_quality: full` — cover hashing fires on ambiguous matches
- 47 fixtures pre-labeled with Metron ids (from existing tags +
  `label_metron.py` cross-references)
- Seed: default (0)

## Execution

Single chunk: 247 fixtures, ~5-7 hours cold-cache wall-clock.

The labeler hit Metron's 20/min rate-limit on its first run and exited
early (false-negative bug — see
[`2026-05-13-metron-series-filter-bug.md`](2026-05-13-metron-series-filter-bug.md)
for the parallel `series` filter bug). Fixed in commit `22650f1`:
labeler now auto-retries on `RateLimitError`. Re-running the labeler
after the fix found 0 additional Metron cross-references — the
remaining 200 fixtures genuinely don't have cv_id cross-references in
Metron's index.

relaunching the 1973 series, etc.). These probe a specific
multi-volume disambiguation failure mode that slimlib's "first issue
of a unique series" sampling underweighted.

## Failure analysis

The 14 CV misses cluster into two distinct patterns:

### Pattern A: Multi-volume "older record wins" (7 cases)

A reissue/relaunch loses to an older "main run" of the same series
because the expected answer **isn't in CV's top-5 search results**
under the FAST budget's max_volumes=5 cap. The matcher's top pick
is then the highest-scoring of what came back — usually the original
run, since its metadata is most complete in CV.

| Comic (year)               | Got (year)              | Year diff |
| -------------------------- | ----------------------- | --------- |
| Akira (2000)               | Akira (1998)            | 2         |
| Ghost in the Shell (2009)  | GitS (1995)             | 14        |
| Savage Sword of Conan (2024) | SSoC (1974)           | 50        |
| Conan the Barbarian (2025) | CtB (1973)              | 52        |
| Storm (2024)               | Storm (2014)            | 10        |
| Powers (2015)              | Powers (2000)           | 15        |
| X-Men #1 Facsimile (2019)  | The X-Men (1963)        | 56        |

**This is NOT a year-scoring bug.** When the expected isn't in the
candidate list, year-decay won't help. The root cause is CV's search
relevance: under `fast` budget (max 5 volumes returned), CV
prioritizes the older / better-known volume for ambiguous queries like
"Akira" or "X-Men #1." The newer reissue/facsimile doesn't make the
top 5.

**Potential fixes** (out of scope for this calibration cycle):
- Increase `_MAX_VOLUMES_PER_SEARCH` for `fast` from 5 → 10 (doubles
  API cost; might help)
- Filter by year *at the search step* (CV's API supports
  `start_year`); not sure if the gain justifies the complexity
- Accept this as a library characteristic — these are reissues that
  share covers with the original, and the matcher genuinely can't
  tell from metadata alone which volume is "right"

### Pattern B: CV catalog duplicates (6 cases)

Two CV records of the same series, same year, different volume_ids.
The matcher's `_apply_tied_metadata_tiebreak` prefers lower volume_id
(canonical), which sometimes matches the user's tag, sometimes
doesn't.

| Comic                                 | User tag (vol_id) | Picked (vol_id) | Cover diff |
| ------------------------------------- | ----------------- | --------------- | ---------- |
| Adventures of Barry Ween 2.0 (2000)   | 25954             | **9615**        | 0.47 (sig) |
| Hilda and the Stone Forest (2016)     | 90261             | **94428**       | 0.41 (sig) |
| Who is Jake Ellis (2011)              | 50971             | **38553**       | 0.41 (sig) |
| Fallen Son: Death of CA (2007)        | 54239             | **18420**       | 0.00       |
| Hawkeye: Freefall (2020)              | 132767            | **123859**      | 0.03       |
| Captain America First Vengeance (2011)| 39936             | **55709**       | 0.12       |

Note the cover-hash signal in the first 3 cases is BIG (0.4+ diff) —
the matcher's pick had near-perfect cover match (≥0.94), the user's
tag had medium match (≤0.59). **The matcher might literally be more
right than the ground truth on these.** Same scan, different CV record;
which one is "canonical" is a judgement call that depends on the
tagging tool used.

Fallen Son and Hawkeye are subtler: cover diffs of 0.00-0.03 are below
the `_COVER_DIFF_NOISE_MARGIN` (0.05), so the tiebreak (lower vol_id)
fires. The user's tags both go to higher vol_id; the matcher prefers
lower. Different tagging-tool conventions.

### Pattern C: No candidates (2 cases)

`Step Aside, Pops: A Hark! A Vagrant Collection` and `Punisher Last
Days Volume 3` simply weren't in CV's top-5 search results. Catalog
search-relevance issue, not a matcher bug.

### The 1 Metron miss

`James Bond M (2018) #001` — expected metron_id=137850, got 137804.
Both from the same Metron series ("James Bond 007 (2018)" series).
Wrong-issue-within-right-series, not a wrong-series miss. The
`series_id` filter is doing its job; the issue-number disambiguation
is the remaining noise. Acceptable.

## What this calibration validates

- **The `series_id` filter fix works.** Metron returns
  series-constrained results; the matcher correctly scores them
  against the profile. 31/31 in auto-write band.
- **The Phase D series-name pre-filter does NOT cause false
  negatives.** None of the 14 CV misses are "pre-filter dropped the
  right volume" cases — the misses are all CV-search-relevance or
  catalog-dupe issues, not pre-filter false negatives.
- **Cover hashing fires correctly when invoked.** The bigmedia data
  shows cover_score values ranging from 0.47 to 1.00, with the
  blended score behaving as designed.
- **Phase E (solo-viable confidence floor) is doing its work.** Only
  1 outcome in the 0.50-0.70 solo-viable band, and it's correct.

## Architectural takeaways

1. **The next high-leverage improvement isn't year-scoring; it's
   search relevance.** The Pattern-A cases (7 of 14 misses) are all
   "right answer not in candidate list" — a year-scoring fix can't
   recover from that. Either:
   - Increase `max_volumes` for the FAST budget specifically for
     ambiguous queries, OR
   - Add server-side year filtering (CV supports `start_year`), OR
   - Add a retry path that broadens the search when no candidate
     clears `min_confidence`

2. **CV catalog duplicates are a real signal worth addressing
   beyond the simple lower-vol-id tiebreak.** When cover-hash signal
   is high (both candidates score ≥ 0.9), maybe trust cover over
   volume_id even at small diffs. Or accept that the user's tagging
   tool convention is what defines "canonical" and document the
   trade-off.

3. **Metron is a viable second source on Big-Two-heavy libraries.**
   97% accuracy on 33 labeled fixtures is enough to justify dual-source
   calibration going forward. Future calibrations on similar libraries
   should label both CV and Metron ground truth.

## Reproducing this run

```bash
# 1. Sample the fixtures:
uv run python -m tests.calibration.sample \
    /Volumes/Media/Comics/ \
    --count 247 \
    --cover-quality full \
    --output tests/calibration/fixtures-bigmedia.json

# 2. Label Metron ids (~17 min worst case at 20/min cap, with
#    auto-retry on rate-limit since commit 22650f1):
uv run python -m tests.calibration.label_metron \
    --fixtures tests/calibration/fixtures-bigmedia.json

# 3. Calibrate. 247 fixtures fits in one chunk (~5-7 hours):
uv run python -m tests.calibration.run \
    --fixtures tests/calibration/fixtures-bigmedia.json \
    --api-budget fast --resume --limit 300

# 4. View cumulative results:
uv run python -m tests.calibration.summarize \
    --fixtures tests/calibration/fixtures-bigmedia.json --misses
```

The fixtures.json output isn't committed (gitignored, per-developer
library paths). Numbers will differ for libraries with different
publisher / decade mixes.
