## What this proves

1. **`_PROMPT_LOCK` works under realistic contention.** The lock is
   class-level (`comicbox/box/online_lookup.py:208`), so all
   `Comicbox` instances in all worker threads share one mutex. With 8
   workers concurrently arriving at PROMPT resolution, every recorded
   `[enter_ns, exit_ns]` interval is disjoint from every other. The
   unit test (`tests/unit/test_run_parallel.py`) verified this in
   isolation; this run verifies it under the real
   `Runner._run_parallel` â†’ `ThreadPoolExecutor` â†’ `_run_one` â†’
   `Comicbox.run()` stack.

2. **Selector overrides work.** The harness monkeypatched
   `comicbox.box.online_lookup.cli_selector` with a recording
   stub. The 23 captured events confirm the production code path
   resolved to the patched selector, not the imported `cli_selector`
   from `comicbox.online.prompt`. This is also the path
   programmatic callers use via `Comicbox(path,
   online_selector=fn)` â€” codex integration plays here.

3. **No deadlocks under 8-way contention.** All 23 selector calls
   completed; the run terminated cleanly. The lock + the matcher +
   the source layer interact correctly under parallel pressure.

## What this doesn't prove

- **Real user think-time is variable.** This run used a fixed 0.5 s
  per prompt. A real user takes anywhere from 1 s (obvious pick) to
  30 s (squinting at the candidate list). The lock will hold either
  way â€” `threading.Lock` is uniform-fair â€” but UX considerations
  like "should we show progress for queued workers?" aren't
  empirically tested here. Out of scope.
- **The `questionary` TTY interaction itself.** The recording
  selector bypasses the actual prompt rendering. A separate manual
  smoke run with the real `cli_selector` in a TTY is still worth
  doing before declaring 4.0.0 fully shipped.

## Harness notes

- The harness drives the real CLI entry point (`comicbox.cli.main`)
  with a constructed argv, not a subprocess. This lets us
  monkeypatch in-process while still exercising the production
  parallel runner end-to-end.
- `--force-search` is essential: without it, fixtures that already
  carry a stored `metron:` or `comicvine:` id take the "refresh via
  stored id" path and never invoke the matcher â†’ no prompts.
- `--api-budget fast` keeps the API budget down so the harness is
  cheap to re-run; the prompt-lock path is exercised the same way
  regardless of budget.
- `always-prompt` policy is the right knob here: it forces every
  candidate into the PROMPT resolution path. With `normal` policy
  the matcher would auto-write most fixtures and only ~10% would
  prompt, which gives less contention to test.
