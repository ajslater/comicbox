# Slimlib 500-fixture calibration — 2026-05-12

Production-scale calibration against a stratified sample of the developer's
17,500-comic slimlib. Validates Phase B's threshold choices and Phase D's
search-breadth caps under a one-per-series, decade+publisher-balanced
distribution that's harder than the original 339-fixture labeled set.

Phase B's set was a hand-curated mix that included multiple issues per
well-tagged series (Watchmen, Sin City, etc.) — many fixtures probed the
same series-discovery code path, inflating the apparent accuracy. The
slimlib sample is intentionally one-per-series, so each fixture is a
distinct probe.

## Fixture set

500 fixtures sampled from `~/Milliways/Comics/slimlib` via
`tests/calibration/sample.py`:

- **One-per-series dedup**: alphabetically-first issue per series name
- **Stratified by (decade, publisher)**: round-robin across buckets so the
  modern-Marvel bulge doesn't dominate the long tail
- **`cover_quality: thumbnail`**: slimlib is one-page dummies with shrunken
  thumbnail covers, so the matcher runs metadata-only (cover hashing
  doesn't fire). Calibrates the metadata signal exclusively.
- **CV ids**: all 500 fixtures (slimlib is CV-tagged)
- **Metron ids**: 11 (2.2% coverage — see `label_metron.py` notes below)

Seed: default (0), for reproducibility.

## Execution

Chunked over multiple sessions via `run.py --resume --limit N`:

- Chunk 1: ~150 fixtures (limit varied based on session length)
- Chunk 2: ~110 fixtures
- Chunk 3+: remaining ~240, until `--resume` reported 0 remaining
- Total wall-clock: ~3-4 days across multiple overnight/work-day sessions

Each chunk used `--api-budget fast` (the budget Phase B validated at 100%
on the smaller labeled set). The harness's `_merge_outcomes_to_partial`
accumulates chunk outcomes into a canonical `fixtures-slimlib.outcomes.json`.

## Results

### Headline accuracy

| Source    | Correct | Wrong | No candidates | Errors | Accuracy on labeled |
| --------- | ------- | ----- | ------------- | ------ | ------------------- |
| ComicVine | 473     | 15    | 11            | 1      | **96.9%**           |
| Metron    | 0       | 0     | 262           | 0      | n/a (no signal)     |

### CV by score band

| Band                          | Correct / total | Accuracy |
| ----------------------------- | --------------- | -------- |
| 0.95-1.00 (very high)         | — / 0           | n/a (none scored here) |
| 0.85-0.95 (auto-write band)   | 471 / 477       | **98.7%** |
| 0.70-0.85 (prompt zone)       | 2 / 11          | 18%      |

The prompt-zone accuracy is *correctly low*: these are cases where the
matcher recognized its own uncertainty and would route to PROMPT in
production. 18% means "if forced to pick, the matcher is mostly wrong
here" — the right design response is to prompt the user, which the policy
does.

The auto-write band's 98.7% is the operationally meaningful number: when
the matcher confidently auto-writes, it's right 471 out of 477 times.

### vs Phase B

| Metric                  | Phase B (339-fixture labeled) | Slimlib (500-fixture sampled) |
| ----------------------- | ----------------------------- | ----------------------------- |
| Budget                  | fast @ 0.7                    | fast @ 0.7                    |
| CV accuracy on labeled  | 100% (338/338)                | 96.9% (473/488)               |
| Auto-write band         | n/a (one wrong was no_cand)   | 98.7%                         |

The 3-point gap is the right direction: slimlib's one-per-series sampling
probes 500 *distinct* discovery paths, while Phase B's set had many
fixtures sharing series (Watchmen × 6, Conan × 20, etc.). The slimlib
number is closer to "production accuracy on a heterogeneous library."

## Failure analysis

### Auto-write band misses (6)

The cases where the matcher confidently picked the wrong answer.
Categorized by failure mode:

| Case                          | Pattern                                                         | Production behavior   |
| ----------------------------- | --------------------------------------------------------------- | --------------------- |
| The Boys 2009                 | Multi-volume same-name; expected at rank 3 (year-drift)         | PROMPT (gap<margin)   |
| Groo: Hell On Earth 2007      | Lone candidate; right vol not in CV's top 5                     | **AUTO_WRITE** (silent) |
| Superman: Secret Identity     | Tied at 0.91 gap=0.00; expected at rank 2                       | PROMPT (gap<margin)   |
| Transmetropolitan #01-#03     | Multi-issue collection; tied at 0.88; both wrong (collection)   | PROMPT (gap<margin)   |
| JLA: World Without Grown-Ups  | Tied at 0.91 gap=0.00                                           | PROMPT (gap<margin)   |
| Wanted Dossier 2004           | Lone candidate; right vol not in CV's top 5                     | **AUTO_WRITE** (silent) |

Only 2 of the 6 "wrong" calibration outcomes are *silent failures* in
production — Groo and Wanted Dossier, both solo-viable cases below the
confidence threshold. The other 4 route to PROMPT via the existing
`disambiguation_margin` (0.10) — the user picks the right answer.

**Phase E (committed 2026-05-12) addresses the 2 silent failures** by
gating NORMAL's `solo_viable` carve-out on a new
`solo_confidence_threshold` (default 0.95 = same as the global confidence
threshold). Solo candidates below the floor now fall through to PROMPT.

### Prompt-zone misses (9)

Cases the matcher correctly flagged as uncertain. These don't represent
production failures — the user gets prompted and picks.

Patterns:
- **Pre-filter near-miss** (Afterschool, Rain, Midlife, Sandman): single
  candidate scoring 0.71-0.84; expected answer not in CV's top-5
- **Multi-volume year confusion** (Conan, Doctor Strange, Batman 2012,
  Killers, Wanted 2005): expected at rank 2-3 in candidate list

### No-candidates (11)

CV's search returned nothing for the file's series name. Mix of:
- **CV catalog gaps** (Wyrd, Halloween Party 2022, Fables - Farewell,
  Black Hammer '45, Herogasm, Planetary #000, Whiz Comics 1940,
  Millennium 1988) — the comic isn't in CV under that name
- **Filename oddities** (Warpsmiths - Ghostdance, Warpsmiths - Cold War,
  33 Captain America v5 10) — unusual filenames the parser didn't
  resolve cleanly to a series name

### Error (1)

`Justice League of America Special #001 (2009)` — transient CV 502
("Unable to parse response"). Would succeed on retry.

## Metron

500 fixtures × Metron source produced 0 gradeable outcomes:
- 262 no_candidates: Metron returned nothing for the series query
- 238 no_expected_id: Metron returned candidates but the fixture had no
  Metron ground-truth id to grade against

`tests/calibration/label_metron.py` cross-referenced the 500 CV ids
against Metron's `cv_id` filter. Result: **11 fixtures had Metron coverage
(2.2%)**, 489 didn't.

This is a library-characteristic finding, not a matcher gap. Slimlib's
heavy indie / Vertigo / pre-IDW / special-edition lean puts a lot of
weight on stuff Metron doesn't catalog. Metron is strong on Big Two
ongoings; slimlib's diversity sits in its weak spots.

**Metron calibration on slimlib is informationally empty.** Metron was
last validated by Phase B against the labeled fixtures (100% accuracy on
the subset with Metron ground truth). A future calibration on a
Metron-friendly library (e.g. one tagged by `metron-tagger`, or heavier
on Big Two) is the right way to re-exercise Metron at scale.

## Methodology improvements that landed

This calibration cycle drove several harness improvements that are now
permanent:

- **`tests/calibration/sample.py`** — stratified one-per-series sampler.
  Inputs: walked library + target count. Outputs: balanced fixtures.json.
- **`run.py --resume`** — skip fixtures already in outcomes file. Lets
  multi-day chunked runs accumulate into the canonical outcomes file.
  Merge semantics, not overwrite.
- **`tests/calibration/label_metron.py`** — cross-reference CV ids
  against Metron's `cv_id` filter to recover Metron ground-truth ids
  without manual lookup.
- **`tests/calibration/summarize.py`** — cumulative report across a
  chunked run's outcomes file.

## Pinned conclusions

1. **`fast` budget is safe to ship as the user-facing default** for
   libraries above some size threshold (Phase C's auto-engagement
   handles this). 96.9% accuracy at fast on a stratified production
   sample, 98.7% in the auto-write band.

2. **Phase B's threshold (0.7) holds.** The slimlib data didn't surface
   any pre-filter false negatives that would motivate loosening to 0.6
   or 0.5. The two pre-filter-near-miss cases (Afterschool, Rain) are
   single-candidate cases where the right answer wasn't in CV at all —
   tightening to 0.75 might be safer but wasn't measured.

3. **Phase D's `_MAX_VOLUMES_PER_SEARCH=5` under fast holds.** No
   evidence that the rank-6-or-later candidates would have changed any
   verdicts — and the cost savings at slimlib scale (17,500 comics) are
   real.

4. **Metron-on-slimlib calibration is not productive.** 2.2% coverage
   gives essentially no statistical power. Future Metron calibration
   needs a Metron-friendly library.

5. **The remaining silent failures (Groo, Wanted Dossier) are solo-viable
   below-threshold cases.** Phase E addresses these via a new
   `solo_confidence_threshold` floor (default 0.95).

## Reproducing this run

```bash
# 1. Sample the fixtures:
uv run python -m tests.calibration.sample \
    ~/Milliways/Comics/slimlib \
    --count 500 \
    --cover-quality thumbnail \
    --seed 0 \
    --output tests/calibration/fixtures-slimlib.json

# 2. Label Metron ids where coverage exists:
uv run python -m tests.calibration.label_metron \
    --fixtures tests/calibration/fixtures-slimlib.json

# 3. Calibrate in chunks until --resume reports 0 remaining:
uv run python -m tests.calibration.run \
    --fixtures tests/calibration/fixtures-slimlib.json \
    --api-budget fast --resume --limit 250

# 4. View cumulative results:
uv run python -m tests.calibration.summarize \
    --fixtures tests/calibration/fixtures-slimlib.json --misses
```

The fixtures.json output isn't committed (gitignored, per-developer
library paths). The methodology is reproducible against any tagged
library; numbers will differ based on library composition.
